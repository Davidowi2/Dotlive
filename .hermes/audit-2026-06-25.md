## The Two Parallel Databases ÔÇö Why This Is Confusing

You have **two completely separate databases** wired into the project:

### Database #1: Neon (Render backend)
- **Schema source**: `dotlive-backend/apps/api/src/db/schema.ts` (Drizzle ORM)
- **Migrations**: `dotlive-backend/apps/api/src/db/migrations/0000_*.sql`, `0001_onboarding_intent.sql`
- **Tables**: `users`, `sessions`, `oauth_accounts`, `user_roles`, wallet/vantage/etc.
- **What's deployed**: Probably **partial** ÔÇö main tables present but Phase 8 columns missing
- **Backend code uses this for**: ALL data (auth, wallet, vantage, ventures, etc.)

### Database #2: Supabase (frontend remnants)
- **Schema source**: `supabase/migrations/*.sql` (13 migration files in tracked copy)
- **Tables**: 22 Supabase tables for builder_profiles, services, orders, reviews, Paystack, Whop courses, dot_id transfer, etc.
- **Backend code uses this for**: **NOTHING** ÔÇö backend has zero Supabase references
- **Frontend code uses this for**: `use-dot-data.ts`, `admin.tsx`, `demo.tsx`, `meetings.tsx`, `investor.tsx`, `sessions.tsx`, `onboarding.tsx`, `join.$code.tsx`, `work.tsx`, `paystack.functions.ts`, `upload.ts`, etc.

**These are TWO DIFFERENT databases with TWO DIFFERENT schemas.** When you go to `/admin`, the code calls Supabase (`use-dot-data.ts`) for builder stats. When you go to `/wallet`, the code calls Render/Neon (`api/wallet.ts`). When you go to `/vantage`, also Render/Neon. The Supabase usage is mostly orphan code from a previous architecture that the project inherited.

**The Supabase client throws on import if env vars are missing:**
```typescript

```

So if you deploy without setting `VITE_SUPABASE_URL` etc. on Vercel, **any code that imports `supabase` will throw**, and TanStack Router will catch it as a route error Ôćĺ blank screen on those routes.
